# #1176 [Story] CodeCompass: 変更頻度分析エンジン（git log集計）

## 本文

## 概要

`git log --numstat` を集計し、直近の変更頻度が高いファイルをランキング形式で抽出する。
CodeCompass の蒸留エンジン（レイヤー1）の最初の柱で、ホットスポット判定の入力データとなる。

## 背景

CodeScene の核心ロジックである「変更頻度 × 複雑度」のうち、変更頻度側の自前実装。
親イシュー #727 のレイヤー1で「自前実装が必須（CodeGraph未対応）」と整理されている部分。

## やること

- `git log --numstat --since=90.days` を実行しファイルごとの変更回数・変更行数を集計する
- 集計結果をランキング形式（変更頻度降順）で構造化データ（JSON）として出力する
- 90日間の期間はオプションで変更可能にする

## 受け入れ条件

- [ ] 任意のリポジトリパスに対して `git log --numstat` ベースの変更頻度集計を実行できる
- [ ] ファイルごとの「変更回数」「変更行数（追加+削除）」を取得できる
- [ ] 変更頻度降順でランキングされた構造化データ（JSON）を出力できる
- [ ] 集計期間（デフォルト90日）をオプションで指定できる

## スコープ外

- 複雑度スコアの算出（→ 別ストーリー）
- ホットスポット判定（変更頻度×複雑度の統合）（→ 別ストーリー）

### 中程度・低リスク指摘
- **Low**: `parseArgs` で `--days` の値を `Number()` に変換する際、非数値が渡されると
  `NaN` になり `git log --since=NaN.days` として渡る。内部ツールであり CLI の利用者は
  開発者自身に限られるため実害は小さいが、バリデーションを足すとより堅牢。
- **Low**: `--out` 指定時の `fs.writeFileSync` は親ディレクトリが存在しない場合に
  例外を投げる（エラーメッセージは明確）。CLI ツールとしては許容範囲。

### 良い点
- `execFileSync` を配列引数で呼び出しており、シェルインジェクションのリスクがない
  （セキュリティ上適切な実装）
- バイナリファイルの `numstat` 出力（`-` 表記）を 0 として扱う処理が正しく実装・テスト済み
