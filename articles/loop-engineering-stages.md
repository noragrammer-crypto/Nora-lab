## はじめに

2026年6月、「Loop Engineering」という言葉がAI開発界隈に広まった。  
発端はPeter Steinberger氏のたった2文のポストだ。

> You shouldn't be prompting coding agents anymore.  
> You should be designing loops that prompt your agents.

この言葉は多くの人に刺さった。だが「ループを設計する」とは具体的に何を意味するのか、なぜそれが重要なのか、既存の解説記事では踏み込めていない部分がある。

**「なぜループが安定するのか」「なぜ人間がループの中にいると限界があるのか」**  
その答えは、制御理論にある。

---

## §0 前提：AIとの共同作業はすでに制御系だ

AIエージェントにコードを書かせるとき、私たちは無意識に「制御系」を構成している。

制御工学の基本は、フィードバックの有無、そしてフィードバックの符号（正負）で系の振る舞いが決まることだ。

### オープンループ制御（フィードバックなし）

入力を受け取り、出力する。それだけ。  
結果がどうだったかを次の入力に反映しない。

```
プロンプト → [LLM] → 出力
               ↑
         （次の入力に反映されない）
```

one-shotプロンプトの繰り返しは、これに近い。  
「なんかいい感じにして」→ 出力→「うーん違う」→ 「もっとこうして」→ ...  
これはフィードバックが**人間の頭の中にしか存在しない**ループだ。

問題は、オープンループ系は**誤差を訂正する機構を持たないため、誤差が蓄積し続ける**ことだ。  
スペックのドリフト、コンテキストの劣化、指示の曖昧さの増大——  
長時間のone-shot対話で感じる「なんか話が噛み合わなくなってきた」はこれが正体だ。  
航法で言えば、センサーによる位置補正を持たない推測航法（デッドレコニング）が時間とともに誤差を蓄積していくのと同じ構造だ。

### フィードバック制御

出力の結果を観測し、目標値との差分（誤差）を計算し、次の入力に反映する。

```
目標 → [差分計算] → [LLM/エージェント] → 出力
           ↑                                  |
           └──────── [観測器/Evaluator] ←────┘
```

これが「ループ」の本質だ。  
テストの通過・不通過、Lintの結果、ビルドの成否——  
機械が観測できる「誤差」があって初めて、フィードバック系が成立する。

ただし、フィードバックには符号がある。

- **負帰還（ネガティブフィードバック）**：誤差を打ち消す方向に補正がかかる。目標に収束し、系は安定する。Loop Engineeringが目指すのはこちらだ。
- **正帰還（ポジティブフィードバック）**：誤差を増幅する方向に補正がかかる。系は発散する。マイクとスピーカーのハウリングはこの典型例だ。

AIエージェントのループでも、正帰還は実際に起こりうる。たとえば実装エージェント自身が「できた」と判定し、その甘い判定をそのまま次の入力に使い続けると、誤りが訂正されずに増幅されていく。§3で述べるEvaluatorの独立性は、ループを負帰還側に保つための設計上の要請でもある。

**ループが安定するのは、フィードバックがあり、かつそれが負帰還として働くから。**  
これが出発点だ。

## §1 Loop in the Human（チェックポイントA）

### 状態の定義

**人間がフィードバック路になっている状態。**

```
イシュー/タスク
      ↓
[エージェント実行]
      ↓
  出力・動作
      ↓
 [人間が観測]  ← ここがボトルネック
      ↓
 人間が判断・修正指示
      ↓
[エージェント再実行]
      ↑_____________|
```

エージェントは動いている。ループも回っている。  
しかし**観測と判断が人間に依存**しているため、ループの速度は人間の処理速度に制限される。  
エージェントが非同期・並列で動き始めると、このボトルネックは致命的になる。

### チェックポイントA：あなたはここにいるか？

- [ ] AIエージェントへの指示を、毎回手で書いている
- [ ] 出力を見て「次何をさせるか」を考える時間が、作業時間の大半を占めている
- [ ] エージェントが止まると、自分も止まる
- [ ] 並列で複数タスクを走らせると、追いかけるのが大変になる

2つ以上当てはまれば、あなたは今Loop in the Humanにいる。

### ここから抜け出すために

最初の一歩は「トリガーの外部化」だ。  
「自分がプロンプトを打つ」のをやめ、**イシューやチケットがエージェントを起動する**構造に変える。  
GitHub IssuesのラベルがClaude Codeを呼び出す、Slackのメンションがエージェントを起動する——  
こういった仕組みを作ることで、人間はフィードバック路から半歩外に出られる。

## §2 Loop on the Human（チェックポイントB）

### 状態の定義

**ループは自走する。しかし停止判定が人間依存の状態。**

```
イシュー投入
      ↓
[エージェント実行]
      ↓
  出力・動作
      ↓
[エージェント自己評価?] ← 弱い、または存在しない
      ↓
 [人間が動作確認] ← まだここにいる
      ↓
  クローズ/差し戻し
```

イシューを投げれば動く。PRも上がってくる。  
しかし「これで合ってるか」の最終判断は、人間の目に依存している。

制御理論で言えば、**観測器（Evaluator）が存在しない、または機能が弱い**状態だ。  
エージェントは「作業した」と言うが、その作業が目標を達成したかどうかを**独立して検証する機構がない**。

### 具体例：SoloXP的な開発フロー

GitHubのIssueをトリガーにClaude Codeが動き、PRを作る。  
CIは回る。しかし：

- 「仕様通りか」の確認は人間がPRを見て判断
- 動作確認は人間が手元で実行
- 「完了」の判定は人間がIssueをクローズすることで行う

このフローは、Loop in the Humanより大幅に効率が上がっている。  
しかし**「人間がレビューしないとループが閉じない」**点で、まだLoop on the Humanだ。

### チェックポイントB：あなたはここにいるか？

- [ ] イシューやチケットを入力すると、エージェントが自律的に動く
- [ ] PRやコミットは自動で生成される
- [ ] 「動作確認」「レビュー」は自分がやっている
- [ ] Evaluator（独立した検証エージェント）が存在しない

上2つがYesで、下2つもYesなら、Loop on the Humanだ。

### ここから抜け出すために

必要なのは**Evaluatorの組み込み**だ。  
「作業したエージェント」とは独立した検証機構が、成功条件を自律的に判定する。

- テストランナーの結果を読んで合否を判定するEvaluatorエージェント
- コードの差分を仕様と照合するレビュワーエージェント
- 実際に動かしてスクリーンショットやログを検証するエージェント

これらがループの中に組み込まれて初めて、人間はループの外に出られる。

## §3 Loop Engineering（チェックポイントC）

### 状態の定義

**Evaluatorと検証系がループ内に組み込まれ、人間はループの設計・改良だけを行う状態。**

```
イシュー投入
      ↓
[エージェント実行]
      ↓
  出力・動作
      ↓
[Evaluatorが自律検証] ← 独立した観測器
      ↓
 合格 → [完了通知] → 人間はオプションで確認
 不合格 → [差分をフィードバック] → [エージェント再実行]
      ↑__________________________________|
```

人間への通知は**例外のみ**だ。  
Evaluatorが「判定不能」「人間の判断が必要」と判断したときだけ、人間にアラートが来る。  
通常フローでは、人間はループの外にいる。

### Evaluatorの独立性がなぜ重要か

制御工学において、観測器が制御器と同一であることは設計上の欠陥だ。  
「自分でやった仕事を自分で採点する」と、バイアスがかかる。  
AIエージェントも同様で、自己評価は往々にして甘くなる。

Zennのino_h氏の記事でも指摘されているように、**実装者と検証者の分離**がループの品質を支える核心だ。  
Claude Codeの`/goal`が内部でHaikuを採点役に使うのも、同じ理由だ。

### チェックポイントC：あなたはここにいるか？

- [ ] Evaluatorエージェントが、実装エージェントとは独立して動いている
- [ ] 成功条件が機械的に判定可能な形で定義されている（テスト通過、Lintクリア等）
- [ ] 人間への通知は例外フロー（Evaluatorが判定不能な場合のみ）
- [ ] 自分の仕事は「ループを改良すること」であり「タスクをこなすこと」ではない

これらがすべてYesなら、あなたはLoop Engineeringの段階にいる。

## 3段階の対応表

| 段階 | ループの主体 | 観測器 | 停止判定 | 人間の役割 |
|------|------------|--------|---------|----------|
| Loop in the Human | 人間 | 人間の目 | 人間が判断 | 実行者 |
| Loop on the Human | エージェント | 人間の目 | 人間が判断 | 監督者・レビュワー |
| Loop Engineering | エージェント+Evaluator | Evaluator（独立） | Evaluatorが判定 | 設計者・改良者 |

## §4 Evaluatorの現実：言うのは3秒、実装は3ヶ月

「Evaluatorを独立させましょう」は簡単に言える。問題は実装コストだ。

### Evaluatorのコスト階層

| Evaluator種別 | 実装コスト | トークン・インフラコスト | 個人で現実的か |
|---|---|---|---|
| テスト通過 / Lint | 低（CI流すだけ） | ほぼゼロ | ✅ 今すぐできる |
| コードレビュー | 中（プロンプト設計） | PR単位なら許容範囲 | ✅ やれる |
| E2Eテスト | 中〜高（Playwright等） | 起動タイミング次第 | ✅ 個人の上限 |
| UIUX評価 | 高（ビジョン+基準設計） | 重い | ⚠️ 予算次第 |
| 負荷試験・攻撃的QA | 高（インフラも必要） | クラウド代も爆発 | ❌ 潤沢予算前提 |

「数百のエージェントが常時監視」という事例はAnthropic社内の話だ。  
個人開発者がLoop Engineeringの現実的な到達点として目指すべきは、**E2Eテストまでを自律Evaluatorで回す**ことだ。それ以上は予算の話になる。

### Evaluator実装のROI判断

重要な設計原則がある。

**EvaluatorのROI = ループの価値 ÷ Evaluatorの実装・運用コスト**

このROIが1を下回るなら、人間がレビューした方が安い。  
逆に言えば、**Evaluatorが格安で実装できる領域が見つかったら、即座にループを閉じる判断をすべき**だ。

テストとLintがその典型で、CIに乗せるだけでEvaluatorとして機能する。  
コードレビューはLLMへの1回の呼び出しで済むため、PR単位のコストは許容範囲に収まる。  
E2Eはセットアップコストはかかるが、一度動けば繰り返しコストは低い。

この「コストが下がった瞬間にループを閉じる」判断の速さが、Loop Engineeringの実践的な競争力になる。

## §5 その先：カスケード制御とコンテンツ工場

ソフトウェアの技術的正しさのループが閉じたとき、次に見えてくるのが**市場価値のループ**だ。

制御工学では、これを**カスケード制御**（二重ループ制御）と呼ぶ。

```
【外側ループ：市場価値】
市場観測（PV・反応・いいね）
      ↓
[ギャップ分析エージェント]
      ↓
次のイシュー/コンセプト生成
      ↓
【内側ループ：技術品質】
[実装エージェント]
      ↓
[E2E Evaluator]
      ↓
[配信・公開]
      ↓
市場観測に戻る ───────────────┘
```

内側のループ（技術品質）が安定していることが前提だ。  
内側が不安定なまま外側を閉じようとすると、系全体が発散する。

この構造はソフトウェア開発だけでなく、コンテンツ生産にも同型で適用できる。  
市場の反応をgap_analyzerの入力に戻せば、コンテンツ工場のフィードバックループが閉じる。  
「何を作るか」の判断までが自律化の射程に入ってくる。

## まとめ

「ループを設計する」とはフィードバック制御系を構築することだ。

- **Loop in the Human**：人間がフィードバック路。並列化すると詰まる。
- **Loop on the Human**：ループは自走するが、停止判定が人間依存。
- **Loop Engineering**：独立したEvaluatorがループを閉じる。人間は設計者になる。
- **その先**：技術品質ループの外側に市場価値ループを重ねるカスケード制御。

現実的な個人開発者の到達目標は**E2EテストまでのEvaluator自走**だ。  
Evaluatorの実装コストが下がった領域から順番にループを閉じていく——この判断の速さが実践的な競争力になる。

「プロンプトを書く」から「ループを書く」へ。  
その先に「ループを設計する」があり、さらにその先に「ループを重ねる」がある。
