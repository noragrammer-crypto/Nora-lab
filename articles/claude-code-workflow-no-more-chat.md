# ClaudeCodeとほぼ会話しなくなった話

前の記事（[Claude Codeに月2万溶かしてる人が今すぐやめるべき3つの習慣](https://note.com/noragrammer/n/n3847721d9dad)）で「ワンタスク・ワンスレッド・ワンPR・ワンクリア」と書いた。あれはどんなワークフローにも使える形で書いたものだ。

じゃあ俺自身はどうやってるか、という話をする。

---

## ルーティンはこれだけ

1. GitHubイシューに**次のゴール**を書く
2. ClaudeCodeに読ませる
3. 複雑そうなら**分解させる**（中ボス程度の粒度まで）
4. スプリント実行
5. 完了報告とPRが来る → **マージ**
6. 残タスクがあればまた実行

以上だ。

---

## ClaudeCodeとほぼ会話しなくなった

チャットで指示しながら直してもらうスタイルを、ほぼやめた。

やるのは**打ち合わせ**のときだけだ。設計や仕様の相談、いわゆる「どうしましょうか」の段階。そこは話す。

ただそこでも、話したら終わりにしない。打ち合わせの内容を**議事録としてイシュー化**してもらう。それをルーティンに乗せる。あとは同じだ。

あと端末とGitHubの差分取りとか、操作系の指示のときもチャットする。それ以外はほぼ不要になった。

---

## コマンドログは見なくていい

ClaudeCodeが作業中に流してくるコマンドログがある。

```
Bash: git add -A && git commit -m "..."
Bash: npm run test
Bash: ...
```

これ、見る意味なくない？

一方で、こっちは読みたい。

「テストプログラム作成してます」「実装始めました」「完了しました」——タイムスタンプつきでイシューにコメントさせてるやつだ。途中で止まったとき、ClaudeCodeがこれを読んで「どこまでやったか」を把握して再開できる。リカバリーにも機能する。

コマンドの実行ログはノイズ。経過・完了報告は資産。その切り分けをシステムに組み込んだ。

---

## なぜこれが良いか

**作業記録がGitHubに全部ある。**

これが一番大きい。何をやったか、なぜそうしたかが、イシューとPRを追えばわかる。

もうひとつ。**コンテキストが圧縮されているから、精霊ちゃんが迷わない。**

タスクが明確で、履歴がGitHubに整理されていると、爆速で終わらせて帰っていく。そういう感じになった。

---

## コードはたまに見に行く

ウンウンで終わり。

気持ちとズレてたなら、イシュー切るだけ。

---

## それって普通の外注と同じじゃない？

エアコン掃除を頼む。終わりましたと言われる。バイクのタイヤ交換を頼む。銘柄どうします？と聞かれる。それこれで、じゃあ注文書書いてくださいと言う。始めまーす、終わりました、料金いくらです、まいどー。

ClaudeCodeとのやり取り、そのまんまだ。

- イシュー = 発注書
- 打ち合わせ = 銘柄どうします？のやり取り
- 経過報告 = 始めまーす
- PR = 終わりました＋請求書
- マージ = まいどー

チャットで会話する代わりに、イシューを切るだけ。

シンプルに見えるけど、**コンテキストエンジニアリングの観点からも筋が通っている。**

チャットログは不要な情報だ。「ちょっとそこ直して」「やっぱりこっちで」のやり取りがコンテキストに残り続ける。精霊ちゃんの探索空間を汚染する。

イシューは違う。目的だけが書いてある。判断の経緯が残る。次のタスクで読み返せる。「なんでこうなってるんだっけ」がGitHubを追えばわかる。

会話を記録に変える。コンテキストを圧縮する。それだけで、全部が変わる。

---

## これがデジタルファーマーのスタイル

細かく会話して管理するのではなく、**土壌（イシュー・GitHub）を整えて、精霊ちゃんに任せる。**

収穫（PR）が来たら受け取る。

そーゆースタイルになりました。
