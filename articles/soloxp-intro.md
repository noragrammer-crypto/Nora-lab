## XP をソロ×AI に適用する

eXtreme Programming（XP）はもともとチーム開発の方法論だ。小さいサイクルで回す・テストを先に書く・継続的にリファクタする。これらの原則は今でも有効だが、「ペアプロ」「チームレビュー」など人手を前提にした実践も多い。

SoloXP はその XP の原則を**ソロ開発 × AI コワーク**に適用したフレームワークだ。

チームの代わりに AI エージェントが複数のロールを担う。ペアプロの相手は Claude になる。そして XP の「小さいサイクル」を極限まで小さくした結果、**Issue 1件がだいたい PR 1件に対応する**粒度になった（厳密な1:1ではなく、Issue 926件・PR 721件で比率はおよそ4:3）。

---

## AI コーディングハーネスとしての立場

GitHub Copilot や Cursor が「エディタで書くときの補完」なら、SoloXP は**プロジェクト全体を回す AI コーディングハーネス**だ。

コードを書く前の Issue 作成から、テスト・実装・ドキュメント・PR 発行まで、タスク単位で AI が一気通貫で処理する。人間のインターフェースは Claude Code だが、その裏で複数のエージェントが役割分担して動いている。

---

## Issue ドリブン: Issue = 仕様 = PR

SoloXP の中心にあるのは「**Issue がすべての起点**」という原則だ。

```
GitHub Issue
  → AI がタスク分解
  → 各タスクを処理
  → PR 発行
  → マージ
```

大きな機能（Story）を書くと、AI の Architect ロールがそれをサブタスクに分解して子 Issue を作る。各子 Issue が独立した PR になる。

結果として **Issue と PR の数が近い値で増えていく**。これは一般的な開発では珍しい。通常は「ブランチにまとめて大きめの PR」になるが、SoloXP では粒度を細かく保つことで：

- レビューが小さく保たれる
- どこで何が起きているかが Issue のコメントで追える
- 途中で止まっても他の Issue に影響しない

---

## 実際のフロー

Issue を作ったら Claude Code で Director を呼ぶだけだ。

```bash
/xp_Director 42
```

あとは自動で回る。

```
Architect  → Story をサブタスクに分解・子 Issue 作成
Tester     → テストスイートを書く（最初は RED）
Implementer→ テストを GREEN にするコードを書く
Auditor    → テスト実行・品質判定
Documenter → 仕様書・リファレンスを生成
Director   → PR 発行
```

各ステップの状態は Issue コメントに書き込まれる。

```
[Tester完了]
[Implementer完了]
[Auditor GREEN]
[PR発行済み #87]
```

進捗はブラウザで GitHub を開けば見える。

## スマホ中心で回るほどになった

SoloXP を続けていくうちに、開発環境が変わっていった。

最終的に**スマホメインで Claude Code Web を使って回す**スタイルに落ち着いた。Issue を書いてタスクを起動し、PR を確認してマージする——これだけなら PC は不要だ。

コードを書くのは AI で、判断をするのは人間。その分業がはっきりしたことで、どのデバイスからでも開発を「続けられる」ようになった。

---

## 5ヶ月半で 721 PR のマージ

SoloXP を本格運用して2026年1月16日から2026年6月27日までの約5ヶ月半で、マージした PR は **721件**。同期間の Issue は **926件**（最新の番号はIssueとPRが同じ番号空間を共有するため #1682 まで進んでいる）。

使っているのは Claude Pro プラン（個人契約）。追加のインフラコストはほぼかかっていない。

Issue と PR の比率はおよそ4:3。これは**ストーリーの粒度を細かく保ち続けた結果**だ。大きな変更をひとつの PR に詰め込まないことで、各 PR のコンテキストが明確になり、AI が処理しやすい単位を維持できている。

## SoloXP が向いている状況

- まとまった開発時間が取りにくい
- Issue やタスク管理を GitHub で完結させたい
- AI に任せる部分と自分で判断する部分を明確に分けたい
- TDD を実践したいが一人で回すのが辛い

逆に、スクラッチで設計を探索しながら書くフェーズや、プロトタイプを素早く試す段階は SoloXP の強みが出にくい。Issue に落とせる粒度になってから使うのが効果的だ。

## Loop Engineering の土台として

2026年6月に広まった「Loop Engineering」という概念――AIに**プロンプトを打つ**のではなく**ループを設計する**という発想――から見ると、SoloXP はすでにその実装のひとつだ。

`Architect → Tester → Implementer → Auditor → Documenter → Director` という流れのうち、鍵になるのは **Auditor が実装エージェントとは独立してGREEN/REDを判定する**こと。これは制御理論で言う「観測器（Evaluator）の独立性」そのものだ。自分の仕事を自分で採点しないからループが安定する。

現状はまだ、PRの最終マージ判断を人間が行う「Loop on the Human」の段階にいる。次に詰めたいのは、Auditorの判定をそのままマージ・クローズまで自律化し、人間への通知を例外時のみに絞る部分――いわゆる「Loop Engineering」段階への移行だ。
