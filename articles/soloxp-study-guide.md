# SoloXP 自律型AI協調開発フレームワーク 総合学習ガイド

SoloXP（Solo eXtreme Programming）は、個人開発者がAIエージェントをチームメンバーとして活用し、開発プロセスを自動化・効率化するためのAIコーディングハーネスです。本ガイドでは、SoloXPのアーキテクチャ、役割、ワークフロー、および運用管理について包括的に解説します。

---

## 1. SoloXPの全体像と設計思想

SoloXPは、従来のeXtreme Programming（XP）の原則を「ソロ開発 × AI」に最適化したフレームワークです。「Issueがすべての起点」という原則に基づき、Issueの作成からテスト、実装、ドキュメント作成、プルリクエスト（PR）発行までをAIが自律的に実行します。

### 基本コンセプト
*   **Issueドリブン:** Issue = 仕様 = PR。ストーリーを細粒度のタスクに分解し、IssueとPRの比率をほぼ1:1に保つことで、レビューのコンテキストを明確にします。
*   **制約の逆転:** 従来の開発では「人間の労働」が制約でしたが、SoloXPでは「AIのトークン消費」や「CPUタイム（計算リソース）」が主な制約となります。
*   **非同期開発:** ユーザー（人間）は移動中や隙間時間にスマホでIssueを作成・承認し、実際の作業はAIがバックグラウンド（GitHub Codespaces等）で実行します。

## 2. システム構成と各コンポーネント

SoloXPは、GitHub環境をベースとした自律エージェントシステムです。

### アーキテクチャ構成要素

| コンポーネント | ファイル・場所 | 役割 |
| :--- | :--- | :--- |
| **GitHub Actions** | `.github/workflows/wake_bugyo.yml` | 新規Issue検知、Codespaceの自動起動、生存確認 |
| **奉行デーモン** | `.devcontainer/bugyo-daemon.sh` | タスク管理者。Issueキューの監視、エージェントセッションの起動・管理、自動シャットダウン |
| **メイドワーカー** | `agent/maid_worker.py` | 実作業者。Issueに基づきPR作成やコメントを行う |
| **devcontainer** | `.devcontainer/devcontainer.json` | 実行環境の定義。起動時に奉行デーモンを自動実行する |
| **tmux** | システム内部 | エージェントの並行実行管理。各タスクを独立したプロセスとして隔離 |

## 3. ロール（スキル）の定義と役割分担

SoloXPでは、AIエージェントが複数の「スキル」を使い分け、チームとして機能します。

### 主要スキル一覧

| スキル名 | 役割 | 主な責務 |
| :--- | :--- | :--- |
| **xp_Director** | 司令塔 | 全スキルの実行順序制御、進行管理。コードは書かず、判断と委譲に徹する |
| **xp_Architect** | 参謀 | イシューの分類（Story/Task/Bug）、タスク分解、サブイシューの発行、既存設計との競合チェック |
| **xp_Tester** | テスト担当 | TDDの実践。Unit/Functional/E2Eテストの作成と実行。実装前にRED状態を確認する |
| **xp_Implementer** | 実装者 | テストを満たす最小限の実装。過度な抽象化を避け、GREENにすることを目指す |
| **xp_Auditor** | 品質監査 | テスト実行結果の分析、品質チェック、サブイシューの完了（GREEN）報告、バグ起票 |
| **xp_Documenter** | 文書作成 | 仕様書（spec）、リファレンス、テスト仕様書等の生成 |

## 4. 標準ワークフロー

SoloXPの標準的な開発サイクルは以下の手順で進行します。

1.  **Issue作成:** 人間がGitHub Issueを作成（[Story], [Task], [Bug] 等のタグを付与）。
2.  **起動:** `xp_Director` が起動し、未処理のIssueを検知。
3.  **計画立案:** `xp_Architect` がIssueを読み、必要に応じてサブイシュー（Task）に分解。
4.  **タスク実行（1タスク1PRルール）:**
    *   `xp_Tester` がテストを作成（RED）。
    *   `xp_Implementer` がコードを実装。
    *   `xp_Auditor` がテストを実行し、品質を確認（GREEN）。
    *   `xp_Documenter` が関連文書を更新。
5.  **完了報告:** `xp_Auditor` が親Issueへ完了報告。
6.  **受け入れテスト:** `xp_Director` が全タスクの完了を確認後、E2Eテストを実行し、親PRを発行。
7.  **人間による最終確認:** ユーザーがPRをレビューし、マージ。

## 5. メトリクス管理と最適化

SoloXPの運用においては、以下の「制約候補」を監視し、効率を最大化する必要があります。

### 重要メトリクス

*   **トークン消費率:** 月間上限に対する残量。プロンプトの効率化やモデル選択（Sonnet vs Haiku）で調整。
*   **CPUタイム消費率:** GitHub Codespacesの稼働時間。アイドル時間の削減、即座シャットダウンの実装が鍵。
*   **Issue完了率:** 1日あたりの処理・マージ件数（目標: 3〜5件/日）。
*   **人間の介入時間:** Issue作成（<5分）やPR確認（<3分）の短縮。

## 8. 記述式・エッセイ問題

1.  **XP原則の適用:** SoloXPが従来のXP（eXtreme Programming）から継承している要素と、AI時代に合わせて「逆転」させた要素について説明しなさい。
2.  **1タスク1PRルールの意義:** SoloXPにおいて、大きな機能を1つのPRにまとめず、細粒度のサブイシューごとにPRを作成することのメリットを、AIの特性（コンテキスト制限等）を交えて論じなさい。
3.  **制約理論（TOC）の応用:** SoloXPにおける「トークン」と「CPUタイム」という2つの制約が、開発戦略にどのような影響を与えるか。リソースが枯渇した場合の具体的な対策を含めて述べなさい。

## 9. 重要用語集（グロッサリー）

*   **奉行デーモン (Bugyo Daemon):** Codespace内で常駐し、Issueキューを監視してエージェントを起動する管理スクリプト。
*   **AllGREEN:** 親ストーリーに紐づくすべてのサブタスクが、Auditorによって品質合格（GREEN）と判断された状態。
*   **depends_on:** タスク間の依存関係を定義するフィールド。先行するタスクがGREENにならない限り、後続タスクは着手されない。
*   **Digital Farmer:** SoloXPの主要ペルソナ。移動中などの断片的な時間を活用し、AIに「耕作（コーディング）」を任せる開発者。
*   **ハーネス (Harness):** 馬具になぞらえ、AIを制御・運用するための枠組み全体を指す。
*   **1タスク1PRルール:** AIのコンテキストを最小化し、品質を維持するために、1つのPRには1つのタスクのみを含めるという運用原則。
*   **即座シャットダウン:** CPUタイムを節約するため、すべてのタスクが完了した瞬間にCodespaceを停止させる仕組み。
